# Dave Design System Reference
## Dror's Assets Viewing Experience - Complete Design Documentation
### February 11, 2026

---

## 1. Opening: What Dave Is

Dave is a client-side web application for viewing and managing digital assets - 3D models, images, videos, audio, fonts, documents, and text files - in a responsive grid layout. It runs entirely in the browser with zero build tools, zero server requirements, and zero tracking. Files never leave the user's machine.

**What makes it different:**
- **No build step**: ES6 modules loaded directly, Three.js via CDN import map
- **No server required**: Works on GitHub Pages as a static site
- **Privacy-first**: No analytics, no cookies, no telemetry, no server-side processing
- **Cloud without compromise**: S3 and Google Drive integration runs entirely client-side using Web Crypto API for SigV4 signing and Google Identity Services for OAuth
- **Personality**: 50 welcome messages, Dangerous Dave easter egg, 14 color themes, whimsical copy throughout

## 2. Design Philosophy

### Core Principles

**Dark-first design**: Dark mode is the default. The rationale: most creative professionals (3D artists, designers, video editors) work in dark environments. Their tools (Blender, Photoshop, DaVinci Resolve) are all dark-themed. Dave should feel native to their workflow, not like a jarring white-background web app.

**No-build simplicity**: Every file is readable as-is. No transpilation, no bundling, no minification. `index.html` has an import map pointing to CDN-hosted Three.js. All source files are ES6 modules with direct imports. This means:
- Any developer can read and modify the code immediately
- No `node_modules` folder (except for server/test dev dependencies)
- No build failures, no webpack config, no Babel plugins
- GitHub Pages deployment is just pushing to main

**Privacy by design**: Files are processed client-side. Cloud credentials are stored in localStorage, not on a server. Pre-signed URLs and blob URLs are used for media rendering. There is no backend that could see, store, or log user files.

**Personality-driven UX**: Dave has a voice. Welcome messages say "I'm Dave. I view assets. Feed me folders." Error states have humor. The easter egg is a full CRT boot animation with matrix rain and a 1990s DOS game. The design communicates: this tool was made by someone who cares, not generated by a committee.

## 3. Color System & Theming

### Architecture

Theming uses **CSS custom properties** applied to `:root` and toggled via JavaScript. This approach was chosen over class-based theming because:
- Instant switching: changing 5 CSS variables updates the entire UI in one repaint
- No JavaScript per-element iteration needed
- Works with existing CSS without refactoring selectors
- Composable: any new component automatically inherits the theme

### The 5 Theme Variables

```css
:root {
  --theme-bg: #121212;        /* Page background */
  --theme-surface: #2a2a2a;   /* Cards, panels, elevated surfaces */
  --theme-text: #e0e0e0;      /* Primary text */
  --theme-border: #444;       /* Borders, dividers */
  --theme-accent: #9b77ff;    /* Buttons, links, active states */
}
```

### Why Purple Accent (#9b77ff)

The default accent color is purple because:
- **Uniqueness**: Most dev tools use blue. Purple stands out without being garish.
- **Dual-mode visibility**: `#9b77ff` (dark) and `#7c5ccc` (light) both maintain sufficient contrast ratios
- **Neutral compatibility**: Purple doesn't clash with file type badge colors (green for models, blue for images, red for audio, etc.)
- **Personality**: Purple is creative, unconventional, memorable

### Theme Persistence

```javascript
// Applied BEFORE first paint to prevent flash
const savedTheme = localStorage.getItem('dave_theme') || 'default';
document.documentElement.setAttribute('data-theme', savedTheme);
// Theme CSS is applied synchronously via inline script in <head>
```

### Theme-Aware Components

All components reference theme variables, not hardcoded colors:
- Scrollbars: `scrollbar-color: var(--theme-accent) var(--theme-surface)`
- Modals: `background: var(--theme-surface); border: 1px solid var(--theme-border)`
- Buttons: `background: var(--theme-accent); color: var(--theme-text)`
- Inputs: `background: var(--theme-bg); border-color: var(--theme-border)`

## 4. Typography & Spacing

### Font Stack

```css
font-family: 'Segoe UI', Roboto, -apple-system, BlinkMacSystemFont, sans-serif;
```

**Why this stack**: Segoe UI is native on Windows (the primary dev platform). Roboto covers Android/Linux. The system font fallbacks ensure zero font loading delay on any platform.

| Element | Size | Weight | Context |
|---------|------|--------|---------|

The spacing is **not** token-based (no 4px/8px scale enforced). It uses contextual values:
- Top bar padding: `8px 16px` (desktop), `6px` (tablet), `4px` (mobile)
- Tile grid gap: `12px`
- Button padding: `6px 8px`
- Modal padding: `20px`
- Section padding: `12px 16px`
- Inspector panel: `12px` internal padding

## 6. Animation Language

### Duration Tiers

| Duration | Use Case | Examples |
|----------|----------|---------|
| **0.15s** | Micro-interactions | Icon color on hover, button background |
| **0.2s** | Small UI changes | Checkbox toggle, modal background |
| **0.3s** | Standard transitions | Panel slide, tile appear, dropdown open |
| **0.4s** | Major state changes | Theme switch (body background + color) |
| **1.0-1.2s** | Dramatic effects | Matrix rain fade in/out |

### Easing Functions

| Easing | CSS | Use Case |
|--------|-----|----------|
| Default ease | `ease` | General transitions |
| Material standard | `cubic-bezier(0.4, 0, 0.2, 1)` | Panels, slides |
| Ease-out | `ease-out` | Elements entering view |
| Ease-in | `ease-in` | Elements leaving view |
| Ease-in-out | `ease-in-out` | Bidirectional movement (panel slide, collapse) |

### Key Animations

**Tile entrance** (`tileAppear`):
```css
@keyframes tileAppear {
  from { opacity: 0; transform: translateY(20px); }
  to   { opacity: 1; transform: translateY(0); }
}
/* 0.3s ease-out, staggered via animation-delay per tile */
```

**Panel slide-in** (`slideIn`):
```css
@keyframes slideIn {
  from { transform: translateX(100%); opacity: 0; }
  to   { transform: translateX(0); opacity: 1; }
}
```

**Inspector panel**: Uses `transform: translateX()` with `0.3s ease-in-out` for the slide transition. When floating, transitions to `position: fixed` with no transform animation.

**Theme switch**: `transition: background-color 0.4s, color 0.4s` on `body` ensures the entire page transitions smoothly when switching themes.

## 8. 3D Inspector UX

### Architecture

```
ModelInspectorPanel (controller)
    |
    +-- GLBInspectorAdapter (model-viewer wrapper)
    |       - Uses Google's <model-viewer> component
    |       - Accesses Three.js internals via model-viewer's scene
    |
    +-- FBXInspectorAdapter (Three.js direct)
            - Custom renderer, camera, controls
            - Direct Three.js scene graph access
```

### Toolbar (Floating, Top-Right of 3D Preview)

```
[Wireframe] [Grid+Axes] [BBox] [Skeleton] [Normals] [AutoRotate]
[Background] [Lighting] [Screenshot] [Inspector Panel Toggle]
```

- **Position**: `position: absolute; top: 12px; right: 12px` inside the viewer wrapper
- **Style**: `backdrop-filter: blur(12px)` with semi-transparent background
- **Icon size**: 42x42px buttons with 18px icons
- **Grid icon**: Custom SVG with 3D wireframe cube + colored axis arrows (R=X, G=Y, B=Z)

### Inspector Panel

- **Default**: Slides in from right edge, 350px wide
- **Resizable**: Left-edge drag handle
- **Floating mode**: Toggle to undock as draggable window
- **Sections** (all start collapsed):
  1. **Stats**: Vertices, triangles, meshes, materials count
  2. **Materials**: Collapsible per-material with property editors (color, roughness, metalness, emissive, opacity, side, transparent). Texture drag-and-drop slots.
  3. **Animations**: List with individual download, select all/none. Transport bar with play/pause, scrub, speed control.
  4. **Helpers**: Wireframe, grid, bounding box, skeleton, normals toggle
  5. **Scene**: Background presets, lighting presets
  6. **Export**: Texture resize dropdown (Original/2048/1024/512/256), simplify slider, animation selection, 3 export buttons (GLB with textures, GLB without textures, textures only)

### Export System

The unified `_doExport()` method applies all modifications non-destructively:
1. Saves original state (textures, geometry, bone transforms)
2. Applies texture resize (if selected)
3. Applies mesh simplification (if slider > 0)
4. Selects animations (all, none, or specific clips)
5. If no animations selected: pauses mixer, waits 2 frames, calls `skeleton.pose()` for bind pose
6. Exports via `GLTFExporter`
7. Restores all original state

**File size estimation** (shown before export):
- Geometry: Sum actual `BufferGeometry` attribute `array.byteLength` values
- Textures: `width * height * 0.75` bytes per texture (empirical average)
- Animations: Sum of all animation track `times.length * values.length * 4` bytes
- Draco detection: Parses GLB JSON chunk for `KHR_draco_mesh_compression`; export always outputs uncompressed

## 9. Performance Patterns

### Lazy Loading

Tiles render placeholder cards immediately, then load actual content (thumbnail, 3D preview) when scrolled into view via `IntersectionObserver`. Blob URLs are created on demand and cleaned up (`URL.revokeObjectURL`) when tiles scroll out of view.

### Web Worker Scanning

Folder scanning runs in a Web Worker (`folder_scanner_worker.js`) to prevent UI blocking during large folder traversals. The worker posts incremental progress updates, enabling the UI to show scanning progress.

### Memory Management

`memoryManager.js` implements queued disposal:
- Tracks created blob URLs, Three.js scenes, textures
- Disposes resources in FIFO order when memory pressure is detected
- Prevents accumulation during long sessions with many folder switches

### CSS Containment

```css
.tile {
  contain: layout style paint;
}
```

This tells the browser that a tile's internal layout changes won't affect siblings, enabling faster repaints during scroll and animations.

### Blob URL Caching

Cloud files use blob URLs (fetched once, cached as `URL.createObjectURL`). Pre-signed URLs (S3) have 1-hour expiry. Google Drive uses object URLs created from `getFileBlob()`.

## 11. Specific Decisions & Why

### Why Dark-First?

**Decision**: Dark mode is the default. Light mode is opt-in via theme selection.

**Rationale**:
- Target users are 3D artists and creative professionals who work in dark-themed tools (Blender, Maya, Substance, Photoshop)
- Asset previews (especially 3D models) look better against dark backgrounds
- Reduces eye strain during extended viewing sessions
- Dark themes are the modern expectation for developer/creative tools

### Why No Build Tools?

**Decision**: Zero build pipeline. No webpack, Vite, Rollup, or TypeScript.

**Rationale**:
- **Simplicity**: Any developer can clone and start editing immediately
- **CDN-powered**: Three.js and Font Awesome loaded from CDNs via import map
- **GitHub Pages compatible**: Push to main, site updates. No CI/CD build step.
- **Debuggability**: Source files in DevTools match source files on disk
- **Trade-off accepted**: No tree-shaking, no minification, no TypeScript safety. For a tool this size, the simplicity wins.

### Why Client-Side Cloud?

**Decision**: S3 and Google Drive integration runs entirely in the browser.

**Rationale**:
- **Static hosting**: Dave must work on GitHub Pages. No server = no server costs, no server maintenance, no server downtime.
- **Privacy**: Credentials stay in the user's browser (localStorage). No server sees their S3 keys or Google tokens.
- **SigV4 in the browser**: Web Crypto API makes this possible. The S3Client implements full AWS Signature Version 4 signing using `crypto.subtle`.
- **Trade-off accepted**: S3 buckets must have CORS configured. Google OAuth must use "Web application" type. These are one-time setup costs.

### Why Purple Accent?

**Decision**: Default accent is `#9b77ff` (dark) / `#7c5ccc` (light).

**Rationale**:
- Blue is overused (every default Bootstrap/MUI/Tailwind app)
- Red/orange imply warning/error states
- Green conflicts with success indicators
- Purple is creative, distinctive, and visible against both dark and light backgrounds
- It doesn't clash with file type badge colors used in the asset grid

### Why CSS Custom Properties Over Class-Based Theming?

**Decision**: 5 CSS custom properties instead of `.theme-dark`, `.theme-light`, etc.

**Rationale**:
- **One line to switch**: `document.documentElement.style.setProperty('--theme-bg', newColor)` updates everything
- **No specificity wars**: Custom properties cascade naturally
- **No JavaScript loops**: Don't need to find and update every themed element
- **Future-proof**: Adding a new theme means adding a new set of 5 values, not new CSS rules for every component
- **Pre-render possible**: Inline script in `<head>` applies theme before first paint, preventing flash

### Why Factory Pattern for Asset Handlers?

**Decision**: `AssetHandlerFactory.create(type)` returns the appropriate handler subclass.

**Rationale**:
- **Extensibility**: Adding a new file type means creating one class and registering it in the factory. No switch statements scattered across the codebase.
- **Encapsulation**: Each handler knows how to render its preview, create its fullscreen view, and handle its interactions
- **Testability**: Handlers can be tested in isolation
- **Note**: The handler system is currently disabled (`const useNewHandler = false`) - legacy code paths are primary. The factory exists for future migration.

### Why Font Awesome?

**Decision**: Font Awesome 6.0.0-beta3 via CDN.

**Rationale**:
- Comprehensive icon library covering all UI needs (files, folders, cloud, settings, media controls)
- CDN-loaded = no build step, no icon font compilation
- Well-known to developers for quick icon lookup
- The beta version was chosen for access to newer icons; stable enough for production

### Why Three.js 0.161.0?

**Decision**: Pinned Three.js version via CDN import map.

**Rationale**:
- Required for FBX loading, GLB inspection/export, and material editing
- Version pinned to avoid breaking changes from Three.js's aggressive deprecation cycle
- CDN-loaded with import map means no `node_modules`, no bundling
- `model-viewer` (Google's web component) handles GLB rendering; Three.js handles FBX + inspector
